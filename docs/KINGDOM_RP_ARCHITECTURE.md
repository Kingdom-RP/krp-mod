## Ключевые типы данных (`data/`)

- `Path` enum — 5 путей: CRAFT, HARVEST, MINING, WAR, MAGIC (с полем `index`)
- `Spec` enum — все специализации с полем `id` (строка для сериализации)
- `BlockEntry(Path path, float xpReward)` — XP за ломку блока → `BlockXPMap`
- `BlockTierEntry(Spec spec, int level)` — требование уровня спец. для добычи
  блока → `BlockTierMap`
- `AnimalTierEntry(Spec spec, int level, float breedXP)` → `AnimalTierMap` —
  гейтинг разведения животных по уровню Фермера + XP за разведение. Ключ —
  `EntityType`. Тот же уровень гейтит доение/стрижку
- `PlantEntry(Spec spec, int level, float plantXP)` → `PlantTierMap` —
  гейтинг **посадки** растений по уровню Фермера + XP за посадку. Ключ —
  блок, появляющийся при посадке (результат `EntityPlaceEvent`).
  `PlantTierMap.isGrowable(block)` — набор растущих культур, которые НЕ
  помечаются в `PlacedBlockTracker` (см. ниже)
- `OreDropMap` — Set<Block>, какие блоки руды (для бонусного дропа шахтёра)
- `KillEntry(Path path, float xpReward)` — XP за убийство моба → `MobKillMap`
- `FishingXPMap` — XP за улов рыбалки (`get(item)`) + флаг сокровища
  (`isTreasure(item)`), path=HARVEST
- `MobDamageMap` — XP за нанесение урона мобу (по тирам, отдельно от убийства)
- `CraftEntry(Path path, Spec spec, float xpReward)` → `ItemCraftMap`. Крафт
  ВСЕГДА успешен — поля `baseChance/maxBonus/k` удалены (ни одна запись их не
  использовала, шанс провала крафта нигде не применялся). Прогрессия — на
  лестницах доступа (`ItemCraftTierMap`/`FoodTierMap`) и эффектах навыков
- `SpecRequirement(Spec spec, int level)` — гейтинг по уровню КОНКРЕТНОЙ
  специализации (не пути). Заменил старый `TierRequirement(Path, level)`:
  крафт/ношение профессионально привязаны к спеку, а не к пути целиком.
  - `ItemCraftTierMap` — `Map<Item, List<SpecRequirement>>`, ограничения на
    **крафт**, выполнены должны быть ВСЕ (поддержка dual-spec: золотая
    морковь/яблоко = Алхимик 3 + Зачарователь 3). Спек — профессия, делающая
    предмет (Кузнец/Инженер/Алхимик/Зачарователь), как в `ItemCraftMap`.
  - `ItemUseTierMap` — `Map<Item, SpecRequirement>` (одно требование),
    ограничения на **ношение/использование**: броня+ближнее оружие = Воин,
    арбалет = Лучник (носить умеет один спек; путь не подходит — иначе лучник
    не надел бы броню по уровню Воина).
  - Один предмет может быть в обоих маппингах с разными спеками (незеритовый
    меч: крафт = Кузнец 7, ношение = Воин 7).

## Система прокачки (PlayerData)

- `pathXP[Path]`, `pathLevel[Path]` — float/int массивы по индексу пути
- `specializationLevels: Map<String specId, Integer level>`
- `MAX_SPEC_LEVEL = 10` — лимит уровня одной специализации
- `getMaxLevel(path) = SpecializationRegistry.getCount(path) * MAX_SPEC_LEVEL`
  — путь не растёт выше суммы максимумов своих специализаций
- `getXPRequired(path) = 100 * 1.5^level` (конфигурируемо: `baseXP`, `xpCurve`)
- `getXPMultiplier(path)` — штраф по приоритету:
  `priority = (кол-во путей с уровнем выше текущего) + 1`,
  `multiplier = 1/priority` (1-й путь=100%, 2-й=50%, 3-й=33%, ...)
  — штраф применяется **в момент начисления XP**, не задним числом
  (нельзя абузить накоплением уровня и поздним вложением очков)
- Очки специализации: 1 уровень пути = 1 очко, 1 уровень любой специализации
  = 1 очко. Свободно распределяются между специализациями одного пути.

## Система начисления XP (XPSystem)

Единая точка: `XPSystem.giveXP(player, Path, amount)` — применяет
`getXPMultiplier`, начисляет, синкает, шлёт полоску прогресса в HUD
(`PacketHelper.sendXPBar` → `XPGainPacket` → `client/XPHudOverlay`) и сообщение
о левел-апе.

**Полоска прогресса в HUD** (`client/XPHudOverlay`, `IGuiOverlay` поверх
`VanillaGuiOverlay.HOTBAR`): появляется сверху экрана при любом начислении XP,
показывает прогресс к следующему уровню текущего пути. Рисуется процедурно в
стиле ванильного XP-бара (рамка + деления-насечки + блик/тень для объёма) —
ванильную текстуру `icons.png` НЕ берём, т.к. её зелёные пиксели нельзя
перекрасить тинтом (`setColor` только умножает) в голубой/жёлтый. Светло-синяя
при обычном начислении, **жёлтая + дефолтный звук левел-апа**
(`SoundEvents.PLAYER_LEVELUP`) при получении уровня. Состояние единое статическое — на экране всегда не больше
одной полоски, новый пакет перезаписывает старую. Держится `DISPLAY_MS` (3 с) с
затуханием в конце. Данные (`currentXP`/`requiredXP`/`level`/`leveledUp`) шлёт
сервер в `giveXP` — клиент сам не пересчитывает прогресс из синка PlayerData.

Источники XP:
- `onBlockBreak` → `BlockXPMap` (с проверкой `PlacedBlockTracker`)
- `onLivingDeath` → `MobKillMap`
- `onLivingHurt` → `MobDamageMap` (урон мобу) + 1 XP/тик за получение урона игроком
- `onArrowHit` (ProjectileImpactEvent) → 2 XP за попадание стрелой
- `onCraft` (`ItemCraftedEvent`) → `ItemCraftMap` (крафт всегда успешен — XP за
  событие; недоступные рецепты нельзя забрать — гейт `SlotMixin.mayPickup`)
- `onFishing` → фиксированный XP, больше за редкий улов
- Зельеварение/зачарование/наковальня — см. `MagicSystem` и Mixin'ы, дают XP
  в `Path.MAGIC`

## Система ограничений (RestrictionSystem + SpecializationEffects)

- `RestrictionSystem.getCraftRequirements(stack)` → `List<SpecRequirement>`
  (`ItemCraftTierMap`); `getUseRequirement(stack)` → `SpecRequirement`
  (`ItemUseTierMap`). Оба по уровню СПЕЦИАЛИЗАЦИИ
  (`data.getSpecializationLevel`), не пути.
- Использование/ношение без уровня — отменяется, броня снимается
  (`LivingEquipmentChangeEvent`), сообщение в чат
- Крафт без уровня — результат ВИДЕН, но изъятие блокируется (`SlotMixin` на
  `Slot.mayPickup`, `isCraftBlocked`, см. раздел Ремесла) + предупреждение в чат на
  попытку. Для dual-spec проверяются ВСЕ требования (`isCraftBlocked` true, если
  хоть одно не выполнено)
- Добыча блока без уровня специализации (`BlockTierMap`) — ломка отменяется
  (`event.setCanceled(true)`), **без** сообщения о начале — только финальный
  результат "блок не сломался"
- `KRPConfig.RESTRICTIONS_ENABLED` — глобальный рубильник всех ограничений

## SpecializationEffects — формат

Один `@SubscribeEvent` на одно Forge-событие, внутри вызывает приватные
проверочные методы по специализациям:

```java
@SubscribeEvent
public static void onBlockBreak(BlockEvent.BreakEvent event) {
    ... общие проверки (клиент, PlacedBlockTracker) ...
    checkTierRestriction(...);  // может .setCanceled() и return
    checkMiner(...);
    checkLumberjack(...);
    checkFarmer(...);
}
```

Аналогично для `PlayerEvent.BreakSpeed`, `LivingHurtEvent`,
`PlayerEvent.ItemCraftedEvent`, `ArrowLooseEvent`.

## Готовые специализации (полный статус)

### Путь Добыча — ЗАКРЫТ ✅

**Шахтёр**
- XP: камень/руды по `BlockXPMap`, твёрдость/редкость → больше XP
  (deepslate-варианты дороже обычных)
- Доступ по уровням (`BlockTierMap`, spec=MINER):
  ур.1 уголь+медь, ур.2 железо+золото, ур.3 редстоун+лазурит+изумруд,
  ур.4 алмаз+обсидиан(+crying), ур.5 nether quartz/gold + ancient debris
- Скорость добычи: `+5% за уровень` (линейно, `PlayerEvent.BreakSpeed`)
- Двойной дроп руды: `level * 0.025` (2.5%/уровень, макс 25% на ур.10),
  только для блоков из `OreDropMap`

**Лесоруб**
- XP за рубку брёвен: ур.0 (дуб/берёза/ель)=2, ур.2 (джунгли/акация/тёмный
  дуб)=4, ур.4 (мангровое/вишня)=8, ур.6 (crimson/warped/грибы)=16
- XP за обтёсывание бревна топором (ПКМ) = 1 (отдельный обработчик
  `PlayerInteractEvent.RightClickBlock` + проверка `AxeItem`)
- Доступ по уровням (`BlockTierMap`, spec=LUMBERJACK) — соответствует тирам XP
- Скорость рубки: `+5%/уровень` — для брёвен И обработанного дерева
  (`isWorkedWood`: блок, чей предмет крафтит Плотник — доски/мебель/двери/
  заборы и т.п.). Лесоруб хорошо добывает дерево в любом виде; набор «деревянных»
  блоков подвязан к крафт-маппингу Плотника (`ItemCraftMap` spec=CARPENTER),
  отдельный список не дублируется
- Сохранение прочности топора: `level * 0.05` (5%/уровень, макс 50%).
  Откат урона через `server.execute` (следующий тик) с клампом `Math.min(before, ...)`
  — `BreakEvent` срабатывает до нанесения урона инструменту, а наивный
  `setDamageValue(-1)` конфликтовал с Unbreaking (топор самочинился). Кламп к
  снимку «до» не уходит ниже исходного значения → стакается с Unbreaking корректно
- Двойной дроп бревна: `level * 0.025` (как у руды)
- Срубка дерева целиком с ур.5: `0.25 + (level-5) * 0.11`
  (ур.5=25%, ур.10≈80%). BFS по соседним брёвнам (10 направлений с диагоналями
  вверх), лимит `MAX_TREE_BLOCKS = 64`, пропускает поставленные блоки,
  выполняется через `server.execute()` в следующий тик

**Лучник**
- XP: 2 за попадание стрелой (убийство стрелой доп. XP НЕ даёт)

### Путь Магия — ЗАКРЫТ ✅

**Алхимик** — ЗАКРЫТ ✅ (простой путь)

1. **Действия для XP** — завершение брю-цикла стойки; крафт магических
   расходников (`ItemCraftMap`, spec=ALCHEMIST, всегда успех): blaze powder=4,
   стекл. бутылка=2, реагенты (ферм. глаз/светящ. арбуз/магма-крем)=3,
   красители=0.5; добыча натуральных цветов и глоустоуна (`BlockXPMap`,
   path=MAGIC, с `PlacedBlockTracker`): мелкие цветы=0.5, высокие=1,
   глоустоун=2. ⚠️ Цветы ПЕРЕНЕСЕНЫ из Промысла (были HARVEST 0.5);
   pink_petals/spore_blossom остались у Промысла.
2. **Маппинг XP варки** (`data/BrewXPMap`): XP за ШТУКУ полученного зелья
   (`PotionUtils.getPotion(result)`), суммируется по непустым бутылкам (1–3).
   Ключ — `MobEffect` результата (long/strong наследуют XP базового эффекта).
   Тиры: база (без эффектов — awkward/mundane/thick)=2, T1 (night vision/
   swiftness/leaping/slow falling/water breathing)=5, T2 (fire res/healing/
   poison/weakness/invisibility/slowness)=8, T3 (strength/regeneration/
   turtle master=resistance/harming)=12; модификатор (long/strong/splash/
   lingering присутствует) = +2 к базе результата. Заменил фиксированные 25.
3. **Эффекты:**
   - **Лестница доступа** (`data/PotionTierMap`) — гейт «НЕ ЗАПУСКАТЬ варку»
     (бутылки/реагент остаются, blaze powder не тратится). Требуемый уровень =
     max(уровень эффекта, long→5, strong→6, splash→7, lingering→9). Эффекты:
     ур.1 night vision/swiftness/leaping; ур.2 water breathing/slow falling/
     fire res; ур.3 healing/poison; ур.4 strength/weakness/slowness;
     ур.5 regeneration; ур.6 invisibility + resistance(turtle master); ур.8 harming.
     long/strong/splash/lingering определяются по registry-key зелья и типу
     предмета (`Items.SPLASH_POTION`/`LINGERING_POTION`).
   - **Активный бонус** — экономия ингредиента: шанс не потратить реагент.
     Снимок ингредиента (`lastIngredientMap`) фиксируется в гейте; при успешной
     варке и удачном ролле `level * 0.05` (ур.10 = 50%) — возврат 1 шт в слот 3.
     (Двойной выход / длительность / усиление — ОТКЛОНЕНЫ.)
4. **Формулы:** success-шанс варки зависит от «запаса» уровня над тиром зелья:
   `base + (1−base)·ScalingFormula(eff, 1.0, 0.3)`, где
   `eff = max(0, уровень_алхимика − требуемый_уровень_зелья)` (требуемый — из
   `PotionTierMap` по результату). Только что открытое зелье (eff 0) = base
   (0.4); зелье ур.1 у алхимика 10 ≈ 84%; harming (ур.8) у алхимика 10 ≈ 63%.
   **Провал** = уничтожаются только зелья (слоты 0–2); ингредиент НЕ трогаем —
   `doBrew` уже снял ровно 1 шт (не весь стак). Экономия — см. выше.
   **XP за варку начисляется и при успехе, и при провале** (одинаково) — см.
   общий принцип в «История решений».

*Реализация гейтинга/атрибуции (ПРОСТОЙ путь — без блокировки слотов):*
- У стойки НЕТ единого «слота результата»: результат считается отдельно для
  каждой бутылки (`PotionBrewing.mix(реагент, бутылка)`). Варка идёт 400 тиков.
- В `serverTick`/`doBrew` игрока НЕТ → owner-а отслеживаем сами.
- `BrewingStandBlockEntity.serverTick` пересчитывает `isBrewable` КАЖДЫЙ тик
  (не по событию смены слота).
- **Гейт**: `BrewingStandMixin.krp$gateBrewable` — `@Redirect` на вызов
  `isBrewable(items)` в `serverTick`. Реплика ванили — `MagicSystem.isVanillaBrewable`
  (через `BrewingRecipeRegistry.canBrew`), затем `MagicSystem.canBrewGate`.
  `brewTime` читается через `BrewingStandAccessor` (`@Accessor`).
- **owner**: `interactorMap` (кандидат, ловится на `RightClickBlock`) →
  на старте варки (`brewTime==0` и гейт прошёл) замораживается в
  `activeBrewerMap`. Пока `brewTime>0` — гейт НЕ пере-проверяется и owner НЕ
  перезаписывается (иначе чужой клик низкого уровня сбросил бы чужую варку).
- **Сообщение** о недоступности — дедуп по хэшу содержимого слотов

2. **Маппинг XP** (`data/EnchantXPMap`): XP за чару = baseXP(rarity)×уровень,

   VERY_RARE=16. Grindstone — только не-проклятые чары (`xpNonCurse`), ×0.5.

3. **Эффекты:**
   - **Лестница доступа** (`data/EnchantTierMap`). Тир редкости: COMMON=0,

     (maxLevel>1) → 7. Тир по действию: книга на столе=3, наковальня=5.
   - **Фильтрация пула на столе** (`EnchantmentHelperMixin` на
     `getAvailableEnchantmentResults`): стол НЕ предлагает чары выше тира игрока —
     из пула кандидатов вырезаются недоступные ДО взвешенного выбора. Это
     устраняет софтлок «вариант предложен, но кликнуть нельзя» (раньше гейт
     ловил это на клике — ресурсы не терялись, но прокачка вставала). Фильтр
     активен только когда генерация идёт со стола: уровень кладётся в
     `EnchantSystem` ThreadLocal вокруг `getEnchantmentList`
     (`EnchantmentMenuMixin`); для лута/жителей/рыбалки ThreadLocal пуст →
     ваниль не трогается. Книга при ур.<3 → пул пуст (стол не предлагает чары на
     книгу). База опущена на 0 (UNCOMMON тоже): «нужда в прокачке» выражается
     шансом успеха и разблокировкой RARE/VERY_RARE/проклятий, а не локаутом базы.
   - **Число вариантов на столе** (`EnchantTierMap.slotCount`): ур.0–2 → 1
     (только верхний, самый слабый), ур.3–4 → 2, ур.5+ → 3. Серверный отказ
     клика по запертому слоту (`EnchantmentMenuMixin` HEAD) + клиентское
     затемнение и блок клика (`client.EnchantmentScreenMixin`).
   - **Активные бонусы** (стол): усиление чары (+1 уровень, `level·0.03`,
     только для предметов — у книг чары в StoredEnchantments), экономия лазурита
     (`level·0.05`), сохранение опыта (`level·0.05`). (Наковальня): скидка на
     уровни (возврат `cost·level·0.05`), бережное зачарование (урон прочности при
     провале на столе убывает с уровнем: ур.0 = 20%, линейно −1.5%/ур. до 5% на
     ур.10), игнор «Too Expensive» (cost≥40) с ур.8
     (`@ModifyConstant` на `createResult`, ordinal 1+2; для стака предметов кап
     не снимается — анти-абуз).
4. **Формулы:** шанс успеха = `base + (1−base)·ScalingFormula(eff,1.0,0.3)`,
   `eff = max(0, уровень − требуемый_уровень)`. Стол `base=0.35`, наковальня
   `base=0.2` (конфиг `enchantTableBaseChance`/`enchantAnvilBaseChance`).
   Низкий уровень → низкий шанс (ур.0 COMMON ≈ 35%, ур.10 COMMON ≈ 84%).
   **Провал:** стол — снимаются только что наложенные чары + урон прочности
   (20% на ур.0 → 5% на ур.10, бонус «бережное зачарование»; книга пропадает
   целиком); наковальня — уничтожается ТОЛЬКО книга, предмет
   остаётся, уровни списываются. XP даётся И при провале (action был разрешён);
   за access-гейт XP нет.

*Реализация:*
- Стол (`EnchantmentMenuMixin`): захват владельца меню — `@Redirect` на

  оставлен как страховка); RETURN ordinal=2 (`return true`) — XP + ролл +
  провал/бонусы. Изменения шлются `broadcastChanges()`.
- Наковальня (`AnvilMenuMixin`): access-гейт на ВХОДЕ — TAIL `createResult`

- Точильный камень (`GrindstoneMenuMixin`): цель — анонимный класс слота

  `player.containerMenu` (слоты 0/1).
- **Объединение одинаковых предметов на наковальне — НЕ реализовано**
  (отложено, неочевидная механика провала)

   нити, ковры, свечи), стекло (панели/цветное/тонированное), бетонная пудра,
   кожа (выделка из крольчьей шкуры, кожаная броня), книги/бумага, глина/керамика
   (глиняный блок, кирпичи, горшок, крашеная терракота, decorated pot), вся
   каменная кладка; обжиг натуральных материалов в печи (`NaturalSmeltMap` через
   `FurnaceResultSlotMixin` → `XPSystem.onNaturalSmelted`).
2. **Маппинг XP** — **XP масштабируется по тиру доступа** (= уровню гейта), чтобы за
   высокоуровневые изделия давали не «копейки»:
   - Крафт (`ItemCraftMap`): ур.0 базовая стройка (текстиль/свечи/стекло-панели/
     бетон/обычная каменная кладка/кирпич/полировка гранита-диорита-андезита/прочие
     блоки)=1 (глина=2); ур.1 кожа/книга=2, кожаные ботинки/шлем=3; ур.2 кожаные
     куртка/штаны=3, крашеная терракота=2; ур.3 decorated pot=4, цветное стекло=2;
     ур.4 узорная отделка (chiseled/mossy)=3; ур.5 дипслейт=4; ур.6 блэкстоун/кварц=5;
     ур.7 призмарин/пурпур/энд-камень=6. XP — за крафт-событие (пачка плит = 1 событие).
   - Обжиг (`NaturalSmeltMap`, за штуку×кол-во): стекло/камень/гладкие+трескающиеся
     варианты=0.5, кирпич/нижний кирпич/терракота/глазур. терракота=1, губка=2.
     **Не гейтится уровнем** (стекло/камень нужны всем рано) — только XP.
3. **Эффекты:**
   - **Лестница доступа к крафту** (`ItemCraftTierMap`, spec=CRAFTSMAN, гейт изъятия
     результата `SlotMixin.mayPickup`, как у Кузнеца) —
     **размазана 1→7** (чем дальше/декоративнее материал, тем выше гейт):
     ур.0 (без гейта) базовая стройка — текстиль/свечи, стекло (панели/тонир.),
     бетон. пудра, обычная каменная кладка (камень/булыжник/песчаник/кирпич/
     незер-кирпич/грязевой кирпич + полировка гранита/диорита/андезита), глина,
     прочие натуральные блоки; ур.1 выделка кожи+книга+кожаные ботинки/шлем;
     ур.2 кожаные куртка/штаны+крашеная терракота; ур.3 decorated pot+цветное
     стекло (витражи); ур.4 узорная отделка камня (chiseled/mossy); ур.5 дипслейт;
     ур.6 блэкстоун+кварц; ур.7 призмарин+пурпур+энд-камень.
     Будущие модовые предметы из натуральных материалов (рюкзаки и т.п.)
     добавляются в эту же лестницу.
     ⚠️ Кожаную броню **носит** Воин (ур.1, `ItemUseTierMap`), а **крафтит**
     Мастеровой — тот же дуал-паттерн, что «незеритовый меч: крафт Кузнец / ношение Воин».
   - **Активный эффект — экономия материала** (`checkCraftsmanEconomy`): шанс
     `min(50%, level·5%)` вернуть 1 ед. натурального ингредиента (кожа/нить/шерсть/
     глина/крольчья шкура) при крафте. Сорт определяем точно (остаток в сетке →
     рецепт), как у экономии древесины Плотника.
   - **Активный эффект — партия сверх нормы** (`checkCraftsmanBatch`): при крафте
     **базового строительного блока** (массовая переработка сырья — glass_pane,
     bricks, stone_bricks, nether_bricks, mud_bricks, sandstone, red_sandstone,
     quartz_block, clay) шанс `min(40%, level·4%)` получить бонусную пачку +2.
     Аналог «партии» Плотника (доски/палки).
   - **Активный эффект — закалка** (`applyCraftsmanTempering`, `CraftsmanTemperMap`):
     прочность свежей кожаной брони растёт по уровню (50% на открытии тира → 100%
     к уровню полной прочности), как у Кузнеца. Ботинки/шлем (unlock 1, full 6),
     куртка/штаны (unlock 2, full 8) — пороги подняты, чтобы эффект был виден на
     протяжении прогрессии (на низком уровне броня заметно изношена). Карта
     расширяема под модовую броню из натуральных материалов.
     ⚠️ Закалка (и Кузнеца, и Мастерового) применяется при **СБОРКЕ результата**
     (`CraftingResultMixin` → `applyTemperingToCraftResult`), а НЕ на `ItemCraftedEvent`:
     при shift-click событие срабатывает уже после перемещения результата в инвентарь
     отдельным стаком (грабли №15), и закалка терялась (broня выходила 100%). Закалка
     стака результата ДО `setItem` чинит оба пути изъятия (клик и shift).
   - ⚠️ Двойной выход крафта у Мастерового УБРАН (был унаследован от Инженера;
     `checkCraftBonus` пропускает CRAFTSMAN).
4. **Формулы:** экономия = `min(0.5, level·0.05)`; партия = `min(0.4, level·0.04)`,
   бонус +2; закалка `quality = clamp(0.5 + 0.5·(level−unlock)/(full−unlock),
   0.5, 1.0)`; доступ — детерминированный гейт по уровню; XP крафта/обжига —
   детерминированы.
   - ⚠️ Каменная кладка `STONE_BRICKS` перенесена от Кузнеца к Мастеровому (была в
     одной register-строке Кузнеца с механизмами).
   - ⚠️ Известная опечатка MCP-маппинга: поле `Items.CUT_STANDSTONE_SLAB`
     (не `CUT_SANDSTONE_SLAB`); в `Blocks` имя корректное. Красный вариант
     `CUT_RED_SANDSTONE_SLAB` — без опечатки.
   - ⚠️ Удалён весь контент бывшего Инженера, не перенесённый в другие навыки:
     редстоун-логика (компаратор/повторитель/датчик дневного света/redstone lamp/
     trapped chest/рычаг/tripwire/target/note block/jukebox), рельсы (все варианты),
     TNT, respawn anchor, conduit, beacon — больше не дают XP и не гейтятся
     (крафтятся как обычная ваниль). Вагонетки/рельсы-XP к Кузнецу НЕ переносились
     (у Кузнеца уже свои вагонетки).

    (`nextInt(100,600) − lureSpeed*100`). Для не-рыбака пишем как есть. Для
    рыбака: задаём собственное время как **центр ± случайное отклонение**.
    Центр линейно убывает: `350 − level*19` тиков, не ниже `LURED_CENTER_MIN=160`
    (ур.10 → 160 тиков = **в среднем 8с без чар**). Отклонение
    ±`LURED_JITTER_FRAC=15%` от центра. Поверх вычитается чара Lure, **ослабленная
    до 3с/уровень** (`LURE_TICKS_PER_LEVEL=60`, ванила 5с): ур.10 + Lure I ≈ 5с,

    до 3с — см. ниже). Нерф Lure глобальный: и не-рыбакам пишется
    `value + (100−60)*lureSpeed` (без чары = ванила).
    **ВАЖНО — перехватываем именно запись, а не
    чтение `lureSpeed`**: `-=` грузит старое значение поля ДО вычисления правой
    части, поэтому redirect на чтение `lureSpeed` (как было сначала) затирался
    обратной записью `-=` и бонус НЕ применялся (см. «Частые ошибки» №10).

    скорость таймеров). Прокачанный рыбак ловит сокровища и в прудах/у берега.

    `Math.round(level * 0.5)` luck (ур.10 ≈ +5, как Luck of the Sea, стакается с
    реальной чарой). В ванильной loot-таблице это снижает вес хлама (`10−2·luck`)

      (печь и так ломается → дроп; абуз — это XP/производство, не обладание).

    производству и XP за готовку. Двойной крафт еды убран (`checkCraftBonus`

    нестакаемость помеченной порции с обычной; баффовую еду добавит будущий
    мод на еду).
  - Общая логика — `system/CookSystem` (не подписчик, библиотека хелперов:
    `canProduce`/`sendRestriction`/`campfireResult`/`onCooked`), дёргается из

    глобальный рубильник `KRPConfig.RESTRICTIONS_ENABLED` (как `RestrictionSystem`)
    — при `false` всё производство еды разрешено.

## Прочая инфраструктура

- `PlacedBlockTracker` — `Set<Long>` (encode через `BlockPos.asLong()`),
  in-memory, НЕ персистентный (сбрасывается при рестарте сервера — осознанное
  решение, повторяет подход оригинального KRP)
- `/krp stats` — для всех; `/krp addxp|setlevel|reset` — `requires(perm 2)`
  (только операторы); `/krp debug` — для всех
- **Данные игрока — Data Attachment** (`registry/KRPAttachments.PLAYER_DATA`,
  `AttachmentType.serializable(PlayerData::new).copyOnDeath()`); `PlayerData`
  реализует `INBTSerializable<CompoundTag>` (методы с параметром
  `HolderLookup.Provider`). Доступ — `player.getData(...)` (всегда не-null) /
  `setData(...)`. `PlayerDataEvents` — только sync клиенту при входе (attach и
  копирование при смерти делает система аттачментов).
- **Сеть** — `network/NetworkHandler` регистрирует пакеты в
  `RegisterPayloadHandlersEvent` через `PayloadRegistrar`; пакеты —
  `record ... implements CustomPacketPayload` со `StreamCodec` (`SyncPlayerDataPacket`,
  `XPGainPacket` — `playToClient`; `ChooseSpecializationPacket` — `playToServer`).
  Отправка — `PacketDistributor.sendToPlayer/sendToServer`.
- Конфиг `KRPConfig` (**ModConfigSpec**, тип SERVER):
  `baseXP=100, xpCurve=1.5` (реально используются в `PlayerData.getXPRequired`),
  `deathPenaltyDuration=6000, deathXpMultiplier=0.5` (дебафф смерти),
  `brewBaseChance=0.4, enchantTableBaseChance=0.35, enchantAnvilBaseChance=0.2`,
  `restrictionsEnabled=true`. ⚠️ `penaltyCoefficient` и `CRAFT_CHANCE_*` удалены
  (не использовались; штраф XP считается switch-таблицей по приоритету).
- `ItemCraftMap` инициализируется **лениво при первом `get()`** (не зависит от
  конфига — после удаления `baseChance` карта читает только `Items.*`). Привязка
  к `ModConfigEvent` и метод `reset()` удалены за ненадобностью.

## История решений и частые ошибки

### Лесоруб: XP при гейте + биом-локаут (2026-06-20)

- **Баг: XP за заблокированную добычу.** Гейт тира (`checkTierRestriction`,
  отменяет `BlockEvent.BreakEvent`) и начисление XP (`XPSystem.onBlockBreak`) —
  РАЗНЫЕ `@SubscribeEvent`-обработчики. При неудачном порядке листенеров XP
  начислялся до отмены. Фикс: гейту (`SpecializationEffects.onBlockBreak`) задан
  `EventPriority.HIGH`, а `XPSystem.onBlockBreak` в начале делает
  `if (event.isCanceled()) return;`. Это покрывает ВСЕ тир-гейты (руды, урожай, логи).
  ⚠️ Грабли: два обработчика на одно событие в разных классах — порядок не
  гарантирован без явного приоритета.
- **Баланс: оверворлд-деревья разгейчены.** Все породы спавн-биомов (джунгли,
  акация, тёмный дуб, мангровое, вишня) переведены в тир-0 в `BlockTierMap` —
  иначе игрок в саванне/болоте/и т.п. не мог добыть местную древесину. Гейт
  оставлен только на незер-стеблях и гигантских грибах (ур.6). XP за все логи
  начисляется как прежде (`BlockXPMap`, путь Добыча).

### Идеи на будущее: статистика и логирование (бэклог, 2026-06-20)

Решено пока НЕ делать, записано на будущее. Хранилище прогресса остаётся
Data Attachment → NBT в `playerdata/` (для одного сервера это ок, см. обсуждение).
Аналитика — аддитивный слой поверх, основное хранилище не трогаем.

- **Логирование прогресса** (просто, независимо от хранилища): дописать вызовы в
  точки левел-апа / `XPSystem.giveXP` / выбора специализации. Лучший формат —
  **append-only event-log** (CSV/JSON: `timestamp, uuid, ник, путь, уровень,
  событие`) — он же фундамент для любой постфактум-аналитики.
- **Статистика по ОНЛАЙНУ** — тривиально и мгновенно: итерировать
  `server.getPlayerList().getPlayers()` + `getData(PLAYER_DATA)`. Команды вида
  `/krp top <path>`, «кол-во онлайн с уровнем > N».
- **Статистика по ВСЕМ (вкл. офлайн)** — слабое место NBT: данные разбросаны по
  `playerdata/<uuid>.dat`, нужно читать все файлы (через `PlayerDataStorage`/
  `getPlayerIo`, ТОЛЬКО async + кэш). Для частых запросов/лидербордов лучше
  **вторичный индекс**: маленький встроенный SQLite/H2 «stats.db», обновляемый на
  logout/левел-апе → «count level>10 / max level / топ-N» = один SQL-запрос.
- Принцип: Data Attachment = источник истины по геймплею; лог/БД = производная для
  статистики. Переезд на сеть серверов/веб позже = локальная замена слоя
  персистентности (load из БД на вход, async-запись на выход), игровая логика не
  меняется.

### Фермер: убран XP за посадку — эксплойт (2026-06-21)

- **Симптом:** «XP за ломку незрелой пшеницы». На деле ломка незрелого корректно
  НЕ даёт XP (`XPSystem.onBlockBreak`: `CropBlock && !isMaxAge` → return;
  подтверждено, `Blocks.WHEAT` = `CropBlock`). XP капал от **посадки**
  (`WorldEvents.onBlockPlace` → `giveXP(plantXP)`): семена возвращаются при ломке
  ростка → посадка→ломка→пересадка = бесконечный фарм.
- **Фикс:** XP за посадку убран (строка `giveXP(plantXP)` удалена; гейт посадки по
  уровню и трекинг клеток сохранены). Награда Фермера — только сбор СОЗРЕВШЕГО
  урожая. Компенсация: HARVEST-XP культур в `BlockXPMap` поднят на +1
  (пшеница 2→3, морковь/картофель/свёкла 3→4, тыква/арбуз/какао 4→5, нарост 6→7).
- `PlantEntry.plantXP` оставлен в данных (историч.), но больше не начисляется.

### Баланс голода (важность Повара / Добычи, 2026-06-20)

### Серверная проверка модов клиента (2026-06-20)

Анти-чит белый список: сервер кикает игрока, если у того есть моды вне
разрешённого набора. Реализовано на хендшейке NeoForge 1.21 (config-фаза):

- `ModWhitelistConfigurationTask` (`ICustomConfigurationTask`) регистрируется на
  каждое подключение через `RegisterConfigurationTasksEvent` (шина мода,
  `ModWhitelist.onRegisterConfigurationTasks`). При запуске шлёт клиенту
  `ModCheckRequestPayload` и НЕ завершается сразу.
- Клиент (config-обработчик `ModCheckRequestPayload`) отвечает
  `ModListReplyPayload` со списком своих `ModList.get().getMods()`.
- Сервер (`ModWhitelist.validateAndProceed`, `enqueueWork`) сверяет с разрешённым
  набором = **моды сервера ∪ `extraAllowedMods`** (конфиг). Лишние моды →
  `context.disconnect(...)`; иначе `context.finishCurrentTask(TYPE)`.
- Пэйлоады зарегистрированы `configurationToClient`/`configurationToServer`,
  **обязательные** (не `optional`) — ванильные/не-Neo и Neo-без-нашего-мода
  клиенты отсекаются хендшейком NeoForge ещё до проверки.
- Конфиг (SERVER): `modCheck.enabled`, `modCheck.extraAllowedMods`.
- ⚠️ Список модов сообщает сам клиент — защита от честных/казуальных клиентов,
  не от глубоко модифицированных. finishCurrentTask/disconnect — в config-фазе,
  на главном потоке (`enqueueWork`).
- Грабли API: `IPayloadContext.finishCurrentTask` бросает вне config-фазы;
  config-пэйлоады используют `FriendlyByteBuf`-кодеки (не `RegistryFriendlyByteBuf` —
  в config-фазе нет доступа к реестрам).

### Пострелизный ретест миграции 1.21 (2026-06-19) — фиксы

- **Замыленные экраны (K / выбор спец.).** В 1.21 `Screen.render` САМ вызывает
  `renderBackground` → `renderBlurredBackground` (блюр-шейдер `processBlurEffect`),
  причём наш `super.render(...)` зовётся ПОСЛЕ собственного рендера — блюр
  пост-обрабатывал уже нарисованный текст (замыливал), а кнопки (рисуются после
  блюра) оставались чёткими. Простой замены вызова `renderBackground` в override
  НЕ хватило. Фикс: `PathScreen`/`SpecializationScreen` переопределяют
  `renderBlurredBackground` и `renderMenuBackground` пустыми, затемнение — своё
  `graphics.fill(...)`. ⚠️ Грабли: переопределять надо именно подметоды, а не сам
  `render`-вызов.
- **Молоко снимало штраф к опыту.** Возвращена защита через `EffectCure`:
  `XPSystem.onEffectRemove` (`MobEffectEvent.Remove`) отменяет удаление
  `DEATH_XP_PENALTY` при `EffectCures.MILK`. (`getCurativeItems()` в 1.21 нет.)
- **Фермер качался на недозрелом урожае.** `XPSystem.onBlockBreak` теперь
  пропускает XP, если `CropBlock` && `!isMaxAge` (как `checkFarmer` для дроп-бонуса).
- **Наковальня врала про требуемый уровень книги.** Сообщение `AnvilMenuMixin`
  показывало статичный `ANVIL_LEVEL` (5) вместо реального `krp$requiredFor`
  (для чар на макс. уровне / проклятий тир = 7). Теперь выводит фактическое требование.
  Заодно убрано ДВОЙНОЕ сообщение: `createResult` выполняется на клиентском И
  серверном меню (одиночная игра) — сообщение шлём только при `!isClientSide()`.
- **Лук: переработка нёрфа.** `BowItemMixin` был сломан (`speedBonus = 20`
  выкручивал ticks в максимум) и дублировал `ArrowLooseEvent` — удалён (и из
  `kingdomrpcore.mixins.json`). Механический бонус скорости зарядки
  (`event.setCharge`) тоже убран: давал рассинхрон визуал/механика и абуз спама.
  Новый перк Лучника — **дальность полёта стрелы** (`onArrowSpawn`,
  `EntityJoinLevelEvent`): скорость стрелы ×(0.5 + 0.1·ур) → ур.0=50%, ур.5=100%,

  стрела (ур.5+) и так проходит через тот же `EntityJoinLevelEvent` → масштабируется.
  ⚠️ Визуальная скорость натяжения лука НЕ меняется (требует клиентского миксина
  на model-property `pull` — не делалось).
- **Закалка распространена на дерево/камень.** Дерево → Мастеровой
  (`CraftsmanTemperMap`, unlock 1/full 3), камень → Кузнец (`BlacksmithTemperMap`,

### Geймдизайн

- **Очки специализации**: 1 уровень пути = 1 очко = 1 уровень любой
  специализации. НЕ растущая стоимость — проверено и отвергнуто (вызывало
  баги с "недоступна кнопка прокачки на N+1 уровне").
- **Штраф XP считается по уровню пути в момент начисления**, не по вложенным
  очкам специализации — иначе абуз (накопить уровни не вкладывая очки в
  "большой" путь, качать остальные без штрафа, потом резко вложить очки).
- **Максимум уровня пути = кол-во специализаций × 10**, а не фиксированное
  число — после изменения состава специализаций пересчитывается автоматически.
- **Крафт: провал = ресурсы сгорают** (не возвращаются) — иначе бесконечные
  бесплатные попытки нивелируют смысл шанса.
- **XP за действие даётся И при провале шанса** (варка, зачарование стола,
  наковальня-книга, шансовый крафт) — столько же, сколько при успехе. Иначе
  прокачка спека «стоит на месте», пока шанс низкий, и игрок не может выбраться
  из ямы низкого уровня. Наказание за провал — потеря РЕСУРСА/предмета, а не
  XP. ⚠️ ВАЖНО: это касается только **шансового провала действия**, которое
  игроку РАЗРЕШЕНО. За **access-гейт** (нет нужного уровня спека вообще —
  `ItemCraftTierMap`, `FoodTierMap`, `PotionTierMap` «не запускать») XP НЕ
  даётся: действие не состоялось, прокачиваться через запрещённый контент
  нельзя.
- **Зачарование: провал = предмет уничтожается**, а не просто "чары не
  легли" — иначе нет риска и кооперация с зачарователем не нужна (пример:
  игрок с Mending легко чинит низкокачественный предмет, поэтому риск должен
  бить по самому предмету, не по прочности).
- **Натуральные блоки**: подход взят из оригинального KRP
  (`SyntheticBlockHolder.kt`) — простая in-memory карта, без персистентности.
  Осознанный компромисс ради простоты.
- **Фермер vs PlacedBlockTracker**: посадка семени проходит через
  `EntityPlaceEvent` → клетка помечалась «поставленной» → сбор СВОЕЙ грядки
  не давал XP/бонусов (работали только дикие культуры). Решение: растущие
  культуры (`PlantTierMap.isGrowable`) не трекаются. Абуза нет — их нельзя
  поставить в спелом виде. Мгновенно-собираемые (тростник/бамбук/кактус/грибы)
  трекаются как обычно (иначе place-break-place на XP добычи).

### Эффект, не снимаемый молоком (curative-items)

- **Молоко в Forge снимает эффекты через `LivingEntity.curePotionEffects(stack)`,
  а не `removeAllEffects()`.** Каждый `MobEffectInstance` хранит список «исцеляющих
  предметов» (`getCurativeItems()`, по умолчанию = ведро молока). `curePotionEffects`
  убирает только те эффекты, для которых `isCurativeItem(stack)` истинно. Чтобы эффект
  НЕ снимался молоком (но молоко продолжало снимать всё ванильное), при наложении
  чистим список: `inst.getCurativeItems().clear()`. Пустой список persist-ится в NBT
  эффекта → иммунитет к молоку переживает релог. Mixin на `MilkBucketItem` не нужен.
  Применено для `death_xp_penalty` (`XPSystem.onPlayerRespawn`).

### Технические грабли (НЕ повторять)

14. **Анонимные классы Minecraft нумеруются по порядку ОБЪЯВЛЕНИЯ в исходнике,
    включая инициализаторы полей.** Слот результата точильного камня — НЕ
    `GrindstoneMenu$3`, а `$4`: поле `repairSlots = new SimpleContainer(2){...}`
    (инициализатор) занимает `$1`, дальше слоты 0/1/результат = $2/$3/$4. Перед
    мизином на анонимный класс проверять номер по байткоду (`javap -p` на
    `Class$N.class`), а не угадывать по порядку в конструкторе.
15б. **МОДИФИКАЦИЯ крафт-результата (закалка прочности) на `ItemCraftedEvent` теряется
    при shift-click** — следствие №15. При shift `quickMoveStack` перемещает результат
    в инвентарь отдельным стаком (через `split`) ДО `onTake`/`ItemCraftedEvent`, поэтому
    правка `event.getCrafting()` не доходит до игрока (предмет выходил на 100% прочности
    при shift, но 50% при обычном клике/Q/свапе). Решение: модифицировать стак результата
    при его **СБОРКЕ** — `@Redirect` на `ResultContainer.setItem` внутри
    `CraftingMenu.slotChangedCraftingGrid` (`CraftingResultMixin`). Тогда и обычный клик,
    и shift (через `split`/`copy`) берут уже изменённый стак; клиент видит корректное
    превью. ВАЖНО: эффекты, которые ДОБАВЛЯЮТ предметы (экономия/партия/двойной выход),
    на `ItemCraftedEvent` работают нормально и для shift — баг только у тех, что МЕНЯЮТ
    сам стак результата.
15. **`quickMoveStack` (shift-click) перемещает результат в инвентарь ДО вызова
    `Slot.onTake`** (в `ItemCombinerMenu`/`GrindstoneMenu` это видно явно). Любая
    логика «решить успех/провал и при провале отобрать результат» в `onTake` для
    shift-click уже опоздала — предмет у игрока. Решение для наковальни
    Зачарователя: access-гейт на ВХОДЕ (`createResult` не создаёт результат), а
    shift-взятие результата при применении книги блокируется отдельным мизином
    на `quickMoveStack` (`ItemCombinerMenuMixin`), чтобы провал считался только
    на обычном клике (где `onTake` успевает обнулить курсорный стак). Та же
    причина, что и №13.
13. **Для МАШИН (печь/блок-сущность) гейтить слот на ВХОДЕ (`mayPlace`), а не на
    выходе.** Гейт на ВЫХОДЕ у машины плох вдвойне: (а) **софтлок** — продукт готов,
    но его некому забрать, машина застревает; (б) **обход разрушением** — сломав
    блок, игрок получает содержимое в дроп мимо гейта. Правильно — гейтить
    **попадание сырья в слот** через `Slot.mayPlace(ItemStack)`: ванильный
    `moveItemStackTo` его уважает, поэтому ОДНА точка покрывает и ручную укладку, и
    shift-click; продукт с недоступным результатом просто не производится.
    - ⚠️ **`Slot.mayPickup` покрывает ВСЕ способы изъятия, включая shift-click.**
      `AbstractContainerMenu.doClick` в ветке QUICK_MOVE проверяет
      `slot.mayPickup(player)` и выходит ДО `quickMoveStack` (проверено по
      исходнику 1.20.1, строки 389–392). Клик/Q/double-click идут через `tryRemove`,
      тоже зовущий `mayPickup`. Поэтому для слота РЕЗУЛЬТАТА КРАФТА (не машина — нет
      софтлока/обхода: ингредиенты в сетке, дропа при сломе нет) `mayPickup`
      достаточно одной точкой — `SlotMixin` (см. гейт крафта). Второй микстин на
      `quickMoveStack` НЕ нужен.
    Реализация для Повара: входной слот печи подменяется на свой подкласс
    `CookGatedInputSlot` через **redirect `new Slot` в конструкторе
    `AbstractFurnaceMenu`** (ловим все `new Slot`: инвентарные — с контейнером
    `Inventory`, оставляем ванильными; единственный не-`Inventory` — слот сырья,
    его подменяем). Игрок кэшируется в `@Unique`-поле, но добыть его непросто:
    **`@Inject` в конструктор ЗАПРЕЩЁН** этой версией Mixin (`@At("HEAD")` падает
    при APPLY: «Found @Inject targetting a constructor»), а **MixinExtras `@Local`
    НЕ на compile-classpath** в Forge 47.x (`com.llamalad7.mixinextras.sugar` не
    виден компилятору). Рабочее решение — ванильный `@Redirect` на ЧТЕНИЕ поля
    `Inventory.player` (`@At("FIELD", opcode=GETFIELD)`): в конструкторе есть
    строка `this.level = inventory.player.level()` ДО создания слотов —
    перехватываем её, кэшируем игрока, возвращаем поле как есть. Второй `@Redirect`
    на `new Slot` использует кэш. Проверено на `runServer` — оба мизина
    применяются без ошибок.
    XP за «успешный результат» вешать отдельно на `checkTakeAchievements` (единая
    точка всех путей изъятия, `removeCount` = забранное кол-во до ваниль-обнуления).
    Аналогично **гейтинг ПКМ-действий (костёр и т.п.) обязан покрывать обе руки**:
    общий гард `event.getHand() != MAIN_HAND` пропускает off-hand, через который
    действие обходится — для таких механик гейт/XP считать ДО гарда, по
    `event.getItemStack()` (стак нужной руки).
12. **Отмена `PlayerInteractEvent.EntityInteract` на сервере без ресинка =
    фантом у клиента.** Клиент предсказывает взаимодействие с мобом (ведро→молоко
    при доении, съеденный корм при кормлении) ДО ответа сервера. Сервер отменяет
    действие (`cancellationResult=PASS`), но НЕ корректирует клиентский инвентарь
    автоматически → у игрока висит фантомное молоко/исчезнувший корм, хотя на
    сервере их нет. Решение: после `event.setCanceled(true)` вызвать
    `player.containerMenu.sendAllDataToRemote()` — принудительный ресинк
    инвентаря. (Симптом из теста: «сообщение об ошибке есть, но молоко появилось».)
    **Тот же баг — надевание брони по ПКМ** (`RestrictionSystem.onItemUse`,
    `RightClickItem`): отмена use без ресинка → броня визуально надета, в
    инвентаре «испаряется» при клике. LMB-путь чинился сам через
    `onEquipmentChange` (переустановка слота), а ПКМ-путь требует явного
    `sp.containerMenu.sendAllDataToRemote()` после `setCanceled(true)`.
0. **Клиентский `@Mod.EventBusSubscriber` без `value = Dist.CLIENT`** падает на
   выделенном сервере: `Attempted to load class .../KeyMapping for invalid dist
   DEDICATED_SERVER` → `ExceptionInInitializerError` в `<clinit>`. Было у
   `KeyBindings` (статическое поле `new KeyMapping(...)`). Любой класс в `client/`
   с MOD-bus подписчиком обязан иметь `value = Dist.CLIENT` (как `ClientEvents`).
   На клиенте баг не виден — всплывает только при `runServer`.

1. **`event.getX() instanceof Y y`** где `getX()` уже возвращает тип `Y` —
   компилятор Java ругается "subtype of pattern type". Решение: просто
   `Y y = event.getX();` без `instanceof`.
2. **Gradle 8.14 несовместим с ForgeGradle 6.x** (`[6.0,6.2)`). Использовать
   Gradle 8.1.1 в `gradle-wrapper.properties`. НЕ соглашаться на предложение
   IntelliJ обновить Gradle.
3. **Mixin в dev-окружении NeoForge**:
   - `mods.toml [[mixins]]` САМ ПО СЕБЕ не работает в dev — нужен
     MixinGradle-плагин (`id 'org.spongepowered.mixin' version '0.7.+'`)
     + блок `mixin { add sourceSets.main, '...refmap.json'; config '...' }`
   - `mixin.env.disableRefMap=true` нужен в run-конфиге
   - После `genIntellijRuns` проверять что VM-опции не потеряли важные флаги
4. **`@Shadow` не работает для полей родительского класса** (например
   `inputSlots` в `AnvilMenu`, объявлено в `ItemCombinerMenu`). Решение —
   отдельный `@Mixin`-интерфейс с `@Accessor` на родительский класс.
5. **`getDamageAfterArmorAbsorb`** в `LivingEntity` — `protected static`,
   нельзя вызвать напрямую. Считать игнор брони через атрибуты
   `Attributes.ARMOR` / `Attributes.ARMOR_TOUGHNESS` цели.
6. **Inject `@At("TAIL")` в метод с несколькими `return`** — срабатывает
   только на ПОСЛЕДНЕМ `return`, не на всех. Если нужен конкретный `return
   true` среди нескольких веток — использовать `@At("RETURN", ordinal=N)`,
   считая `return`'ы по порядку в декомпилированном байткоде.
7. **Конфиг-зависимые маппинги** нельзя инициализировать в статическом блоке
   если используются значения из `ForgeConfigSpec` — конфиг ещё не загружен на
   этапе class-loading. Решение: ленивая `init()` (+ пересчёт на
   `ModConfigEvent`, если значения могут меняться в рантайме). ⚠️ Сам
   `ItemCraftMap` БОЛЬШЕ не конфиг-зависим (после удаления `baseChance`) —
   привязка к `ModConfigEvent` снята; принцип остаётся для будущих маппингов.
8. **Bonus drop / fellTree через `playerDestroy` внутри обработчика
   `BlockEvent.BreakEvent`** — может рекурсивно вызвать события. Для
   "срубки дерева" используется `server.execute()` (следующий тик) чтобы
   не конфликтовать с текущим событием.
10. **`@Redirect` на ЧТЕНИЕ поля внутри составного присваивания `field -= expr`**
    не даёт изменить это поле из хендлера. JVM грузит старое значение `field`
    на стек ДО вычисления `expr` (где срабатывает redirect), а потом
    записывает `старое - результат`, затирая любую запись `field`, сделанную
    внутри хендлера. Симптом: значение «возвращается» к исходному сразу после
    установки. Решение — перехватывать **запись** поля (`@Redirect` с
    `opcode=PUTFIELD`), а не чтение. Пример: ускоренный клёв в `FishingHookMixin`
    (строка `timeUntilLured -= lureSpeed*20*5`).
9. **PathScreen / SpecializationScreen с динамическими списками кнопок** —
   считать высоту окна и Y-позиции кнопок ДО рендера (в `init()`/helper-
   методе), переиспользовать те же расчёты в `render()`. Раньше было
   дублирование и рассинхрон.
11. **`@Shadow` поля, объявленного в РОДИТЕЛЬСКОМ классе, при `remap=false`
    и без refMap не находится** (ошибка `@Shadow field X was not located in
    the target class`). Пример: поле `random` объявлено в `Entity`, а миксин
    нацелен на `FishingHook` — `@Shadow protected RandomSource random` падает
    при apply. Решения: (а) не шедоуить, а получить значение иначе (в
    `FishingHookMixin` джиттер берём из уже-случайного `value` через
    `Math.floorMod`); (б) если поле всё же нужно — `@Accessor`-интерфейс на
    родительский класс (как для `inputSlots`, см. №4). Поля самого целевого
    класса (`timeUntilLured`, `lureSpeed`, `luck`) шедоуятся нормально.
